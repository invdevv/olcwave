# Устранение неполадок

Реальные проблемы, с которыми вы можете столкнуться, и способы их исправления. Начните с просмотра логов:

```bash
docker compose logs -f api        # backend
docker compose logs -f caddy      # reverse proxy
docker compose logs -f postgres   # database
docker logs olcwave-<tag>-<uuid>  # конкретный OLCRTC контейнер
```

---

## Docker: permission denied

**Симптом:** логи API показывают что-то вроде `PermissionError: /var/run/docker.sock`, либо панель не может получить список контейнеров или запустить их.

**Причина:** контейнер API не имеет доступа к Docker socket хоста.

**Исправление:**

* Убедитесь, что socket подключён - в `docker-compose.yaml` должен быть указан маппинг:

  ```yaml
  /var/run/docker.sock:/var/run/docker.sock
  ```

* На хосте socket должен быть доступен. Если вы запускаете Compose от пользователя без root-прав, этот пользователь должен быть добавлен в группу `docker`:

  ```bash
  sudo usermod -aG docker "$USER"   # затем выйдите из системы и войдите снова
  ```

* В системах с SELinux может потребоваться добавить `:z`/`:Z` к монтированию или настроить SELinux policy.

## Контейнер не запускается

**Симптом:** `POST /containers/run` завершается ошибкой или контейнер сразу завершается.

**Проверяйте по порядку:**

1. **Лимит трафика** - если ответ:

   ```
   403 traffic_limit_exceeded
   ```

   владелец пользователя превысил квоту. Сбросьте или увеличьте лимит:

   `Users page → edit → Reset traffic / change limit`.

2. **Сборка образа** - первый запуск собирает образ `olcrtc`, который клонирует и компилирует OLCRTC из GitHub. Для этого нужен интернет и процесс занимает некоторое время.

   Следите за логами во время первого запуска:

   ```bash
   docker compose logs -f api
   ```

3. **Неверная конфигурация** - профиль может успешно пройти проверку YAML, но быть невалидным конфигом OLCRTC. В таком случае контейнер запустится и завершится с ошибкой.

   Проверьте его логи:

   ```bash
   docker logs olcwave-<tag>-<uuid>
   ```

   EntryPoint требует переменную окружения `CONFIG`. Панель всегда устанавливает её, поэтому если вы видите:

   ```
   CONFIG env is required
   ```

   значит контейнер был запущен не через панель.

4. **Конфликт имени** - имена контейнеров должны иметь формат:

   ```text
   olcwave-<tag>-<uuid>
   ```

   с ровно двумя разделителями `-`.

   Символ `-` внутри `tag` профиля ломает разбор владельца. Переименуйте `tag` профиля.

## Ошибка подключения к базе данных

**Симптом:** логи API показывают ошибки подключения/аутентификации `asyncpg`, либо API постоянно перезапускается.

**Причины и исправления:**

* **Несовпадение данных доступа** - значения:

  ```
  POSTGRES_USER
  POSTGRES_PASSWORD
  POSTGRES_DB
  ```

  в `backend/.env` должны совпадать с теми же значениями в корневом `.env`.

  Если они отличаются, база была инициализирована с одним набором данных, а API пытается подключиться с другим.

  Сделайте их одинаковыми, затем:

  ```bash
  docker compose down
  ```

  Если база уже была создана с неправильным паролем, удалите её volume:

  ```bash
  docker volume rm olcwave_postgres_data   # удаляет все данные
  ```

  После этого:

  ```bash
  docker compose up -d
  ```

* **Неверный host** - внутри Compose:

  ```text
  DB_HOST
  ```

  должен быть:

  ```text
  postgres
  ```

  (имя сервиса).

  При запуске разработки напрямую с хоста должно быть:

  ```text
  localhost
  ```

* **Postgres ещё не готов** - API использует `depends_on` с healthcheck, поэтому это редко случается. Но если API был запущен отдельно, подождите немного, пока Postgres стартует.

## Ошибка подключения к Remnawave

**Симптом:** каждый запрос подписки возвращает `404`, либо API показывает ошибки подключения/таймауты при обращении к Remnawave.

**Исправление:**

* `RW_API_URL` должен быть правильным **API base URL Remnawave** и быть доступным из контейнера API.

  Образ API основан на Python, поэтому можно проверить через Python:

  ```bash
  docker compose exec api python -c "import os,urllib.request; print(urllib.request.urlopen(os.environ['RW_API_URL']).status)"
  ```

* `RW_API_TOKEN` должен быть действующим.

  Отозванный или неверный токен приведёт к тому, что Remnawave отклонит запросы и пользователь не будет подтверждён.

* Помните: `404` от:

  ```
  /sub/{uuid}
  ```

  также является нормальным ответом для UUID, которого Remnawave не знает.

  Проверяйте с UUID, который точно существует.

## Подписка не создаётся

**Симптом:**:

```
GET /sub/{uuid}
```

возвращает бесполезный результат, либо контейнеры не появляются.

**Проверьте весь процесс (см. [architecture.md](architecture.md)):**

1. **UUID существует в Remnawave?**

   Если нет → `404`.

   Это самая частая причина.

2. **Превышен трафик?**

   Тогда вы получите заглушку:

   ```
   Traffic limit Exceeded
   ```

   и HTTP `403`, а не настоящие конфигурации.

   Сбросьте трафик.

3. **Есть ли созданные профили?**

   Если профилей нет - запускать нечего.

   В пакете не будет location.

   Создайте хотя бы один профиль.

4. **Работает ли Docker?**

   Создание контейнеров требует рабочего Docker socket.

   См. раздел про `permission denied`.

5. Следите за логами API во время открытия URL подписки - ошибки генерации конфигурации или `docker run` будут видны там.

## Трафик не обновляется

**Симптом:**:

```
traffic_used_bytes
```

остаётся равным `0`, хотя контейнер передаёт трафик.

**Проверки:**

* Трафик учитывается только для **запущенных** контейнеров:

  ```
  olcwave-*
  ```

  Остановленный контейнер ничего не сообщает.

* Трафик проходит через SOCKS proxy контейнера - учитываются только байты, прошедшие через этот proxy.

  Если клиент реально его не использует, считать нечего.

* Посмотрите сырую статистику внутри контейнера:

  ```bash
  docker exec olcwave-<tag>-<uuid> cat /var/lib/olcwave/stats.json
  ```

  Если здесь:

  ```json
  total_bytes
  ```

  увеличивается, но значение пользователя `used` не меняется, значит цикл сбора не видит данные - проверьте логи API.

* Цикл запускается каждые:

  ```
  TRAFFIC_COLLECT_INTERVAL
  ```

  секунд.

  Если значение слишком большое, обновления будут медленными.

  Уменьшите его и перезапустите API.

* Имя контейнера должно корректно определять владельца:

  ```
  olcwave-<tag>-<uuid>
  ```

  Неверное имя означает, что изменение трафика будет отброшено.

## Проблемы Caddy / SSL

**Симптом:** браузер показывает ошибку сертификата или сайт недоступен по HTTPS.

**Проверки:**

* **Порт 443**

  `docker-compose.yaml` должен публиковать оба порта:

  ```yaml
  ports:
    - "80:80"
    - "443:443"
  ```

  Если вы изменяли Compose и убрали `443`, добавьте его обратно в сервис `caddy`.

* **DNS**

  Домены в:

  ```
  caddy/Caddyfile
  ```

  должны указывать на публичный IP этого сервера.

  Caddy не сможет получить сертификат для имени, которое не указывает на него.

* **Firewall**

  Порты:

  ```
  80
  443
  ```

  должны быть открыты.

  Let's Encrypt проверяет доступность через порт 80.

* **Смотрите логи Caddy**

  ```bash
  docker compose logs -f caddy
  ```

  Там явно отображаются ошибки получения сертификатов:

  * лимиты запросов;
  * ошибки DNS;
  * проблемы валидации.

* **Frontend не может подключиться к API**

  Если SPA загружается, но все запросы падают:

  * проверьте `VITE_API_URL`;
  * проверьте proxy Caddy.

  Caddy удаляет `/api`, поэтому:

  ```text
  VITE_API_URL
  ```

  должен заканчиваться на:

  ```
  /api
  ```

  например:

  ```
  https://panel.example.org/api
  ```

  См. шаг 8 в [installation.md](installation.md).

## Изменения `.env` не применяются

* **`backend/.env`**

  Передаётся в контейнер API через `env_file` и читается при старте контейнера.

  Для применения изменений перезапустите:

  ```bash
  docker compose up -d api
  ```

* **`frontend/.env` (`VITE_*`)**

  Встраивается в собранный JavaScript.

  Нужно пересобрать frontend:

  ```bash
  cd frontend && npm run build && cd ..
  docker compose restart caddy
  ```

* **Корневой `.env` (данные PostgreSQL)**

  Используется только при первом создании volume PostgreSQL.

  Если изменить его позже, потребуется сбросить volume (см. раздел про базу данных).
